1. Цель и контекст
Создать MVP платформы IntegrityOS — системы визуализации, хранения и анализа данных обследований магистральных трубопроводов.
MVP демонстрирует единый цикл:
* Импорт CSV-файлов с данными.
* Отображение объектов и диагностик на карте.
* Список объектов с фильтрацией по методам контроля.
* Карточка объекта (история диагностик, оценка ИИ, параметры).
* Дашборд (статистика по методам, рискам, объектам).
* Модуль AI/ML для классификации критичности.
* Генерация краткого отчета в HTML/PDF.
Система не претендует на промышленное качество.


2. Состав данных
2.1 Таблица объектов контроля (Objects.csv)
Поле
	Тип данных
	Описание
	object_id
	int
	Идентификатор объекта
	object_name
	string
	Наименование оборудования или участка трубы
	object_type
	enum
	crane / compressor / pipeline_section
	pipeline_id
	string
	Идентификатор условного трубопровода
	lat
	float
	Географическая широта
	lon
	float
	Географическая долгота
	year
	int
	Год выпуска / ввода в эксплуатацию
	material
	string
	Марка стали или материала
	

2.2 Таблица результатов диагностик (Diagnostics.csv)
Поле
	Тип данных
	Описание
	diag_id
	int
	Идентификатор записи
	object_id
	int
	Ссылка на объект
	method
	enum

	date
	date

	temperature
	float
	Температура воздуха
	humidity
	float
	Влажность
	illumination
	float
	Освещенность
	defect_found
	bool
	Признак дефекта
	defect_description
	string
	Описание дефекта
	quality_grade

3. Функциональные требования
3.1 Импорт и валидация
* чтение CSV/XLSX;
* мэппинг колонок;
* проверка: object_id, дата, method;
* лог ошибок.
3.2 Хранилище
База данных любого типа.
Обязательные сущности:
* Pipelines
* Objects
* Inspections
* Defects
3.3 Картографическая визуализация
* карта Казахстана;
* магистрали MT-01 / MT-02 / MT-03 (синтетические координаты из исходного ТЗ);
* точки дефектов, цвет критичности;
* фильтры: метод, дата, диапазоны параметров;
* всплывающие карточки.
3.4 Поиск и фильтры
* по объекту/трубе;
* по дате;
* по типу дефекта/методу;
* сортировка по глубине.
3.5 Карточка объекта/дефекта показывает:
* параметры (depth, length, width);
* источник обследования;
* исходный файл (опционально);
* историю по годам.
3.6 ML/AI классификация
Алгоритм критичности:
* ML-модель (DecisionTree/RandomForest/logistic regression)
* Пороговая rule-based модель.
Вывод:
* метка normal / medium / high
* вероятность (если ML)
3.7 Дашборд
Обязательные виджеты:
* распределение дефектов по методам
* распределение по критичности
* топ-5 рисков
* количество обследований по годам
Опционально:
* диаграмма ремонтов
* динамика дефектов
3.8 Отчёты
Генерация документа:
* общая статистика
* таблица дефектов
* рекомендуемые раскопки (по критичности)
* карта участка
Формат: PDF или HTML.


4. Нефункциональные требования
* минимальная производительность: импорт ≤ 20 сек;
* визуализация списка дефектов до 10 000 точек;
* UI дружелюбный


5. Технологии (по выбору участников)
Frontend:
* React / Vue / Angular
* Leaflet / Mapbox / OpenLayers
Backend:
* FastAPI / Flask / Django
* Node.js / Spring Boot
DB:
* PostgreSQL / SQLite / MongoDB


6. Что сдаёт команда
* MVP (репозиторий + инструкция запуска)
* Презентация решения (5–7 слайдов)
* Демо-видео (по желанию)
7. Критерии оценки
* Все данные в рамках кейса являются синтетическими и не отражают реальные показатели технического состояния оборудования или трубопроводов.
* Участники могут дополнять данные собственными тестовыми значениями в пределах предложенной структуры.
* Формат реализации не ограничен: допускается использование любых библиотек, языков программирования и фреймворков.
